# 项目定位与目标  
本项目定位为一个练手型外卖点餐系统，主要目的是帮助初学者通过完整实现前端（微信小程序、商家后台）、后端和数据库的串联，掌握全栈开发流程。功能只需覆盖核心交易闭环（点餐、下单、支付模拟、订单管理），不涉及企业级复杂性。采用 MVP（最小可行产品）思想，只实现核心功能。技术选型固定：后端 Spring Boot、数据库 MySQL，前端用户端使用 UniApp（部署为微信小程序），商家端使用 Vue。我们将使用单体架构，将业务逻辑集中在一个代码库中，以降低开发和部署复杂度；前后端通过 HTTP/RESTful API 通信。

# 需求范围与边界  
- **用户端主要需求**：实现微信小程序登录/注册、首页展示（轮播、广告、自提/外卖入口）、商品分类浏览、商品详情及下单（选规格、加料、杯型等）、购物车、多商品结算、订单结算（选择自提/外卖、地址填写、总价计算、支付确认）、订单列表与详情、再来一单、用户信息与地址管理、退出登录等。  

- **商家端主要需求**：实现管理员登录（无需注册），后台侧导航包含「订单管理」「库存管理」「历史订单」；订单管理页列出待处理订单，可完成/取消订单（并据此改变库存）；库存管理页可搜索、新增/编辑商品（含图片、名称、简介、分类、价格、库存、上架状态等字段）；历史订单页显示已完成和已取消订单列表。  

- **明确的非功能性限制**：

  \- 不使用 Redis、消息队列、微服务、分布式事务等高级技术，不接入真实支付，只做支付流程模拟。
  \- 用户端和商家端采用简单 JWT 登录认证；JWT 仅由后端签发并由客户端保存，不在数据库中存储 token，不实现刷新令牌（Refresh Token）和复杂会话管理。
  \- 不做多门店、多用户权限、推送通知、实时排队、营销活动、优惠券等功能。
  \- 聚焦于逻辑完整、可运行的最小系统（MVP）。

# 核心功能与优先级  
- **本期必须实现**：用户登录/注册、首页展示、分类点餐与商品浏览、购物车管理、下单结算流程（自提/外卖切换、地址填写）、订单生成与状态查询；用户订单列表及详情（显示商品、规格、价格、时间、订单状态等）、再来一单；用户个人中心（显示基本信息、管理地址、退出登录）。商家端包括管理员登录、订单管理（显示订单详情、点击完成/取消）、库存管理（商品的增删改查）、历史订单查看等。以上功能构成最小闭环，是 MVP 的核心。  
- **后期可选功能**：涉及复杂度较高的功能可以后置或简化，例如：**外卖配送流程**（物流跟踪、配送员派单）、**用户评价/评分**、**优惠券/促销**、**多店铺/多门店支持**、**第三方登录/微信支付对接**、**实时消息推送**等。若在开发中发现部分功能超出初学者范围（如复杂的选项配置、多种支付方式等），可先剔除或统一简化。例如将“立即购买”直接视为单品下单，支付后直接生成订单即可，无需处理库存锁定等复杂逻辑。  

# 前后端交互逻辑  
前后端分离，统一通过 RESTful API 交互。**用户端（微信小程序）**通过 HTTP 请求调用后端接口，例如：用户登录接口 `/api/user/login`、注册 `/api/user/register`，获取商品分类和列表 `/api/products`，添加购物车 `/api/cart/add`，查看购物车 `/api/cart/list`，提交订单 `/api/order/create`（包括订单详情、地址、自提/外送方式等），查询订单列表 `/api/order/list`，订单详情 `/api/order/{id}`，管理地址 `/api/address/**` 等。用户端与后端交换 JSON 格式数据，**状态验证采用简单 JWT 登录认证。后端登录成功后签发 JWT，前端负责本地保存并在请求头携带；数据库不存储 JWT，不实现刷新令牌和复杂会话管理。**后端登录成功后签发 JWT，前端负责本地保存并在请求头携带；数据库不存储 JWT，不实现刷新令牌和复杂会话管理。**商家端（Web）**则调用后台管理接口：管理员登录 `/api/admin/login`，获取待处理订单列表 `/api/admin/orders/pending`，完成或取消订单 `/api/admin/orders/{id}/complete`、`/cancel`，查看已完成/已取消订单 `/api/admin/orders/history`，获取商品列表 `/api/admin/products`，新增/编辑商品 `/api/admin/products/create`、`/update` 等。这些接口由 Spring Boot 提供，统一返回 JSON 结果，前端 Vue 通过 Axios 或类似库调用并更新页面状态。整个系统遵循简单的请求-响应流程，无需复杂的异步消息或服务间调用。  

# 页面结构与跳转关系  
**用户端**页面按 UniApp/微信小程序模式组织：启动进入登录/注册页，登录成功后进入主界面（底部 4 个 tab）。4 个主 tab 分别为：  
- **首页**：仅展示轮播图、广告、自提/外卖入口等非交易内容，点击后不含交易逻辑。  
- **点餐**：左侧为固定分类列表，右侧为当前分类下商品列表。商品卡片显示图片、名称、简介、价格。点击商品卡片进入商品详情页，可选择规格（甜度、温度、加料、杯型等），页面底部显示当前选择的总价和数量，有“立即购买”“加入购物车”按钮。点击“加入购物车”后，页面底部出现一个浮动购物车区域，显示购物车总价与“结算”按钮；当购物车为空时该区域自动隐藏。购物车可容纳多个商品。点击“结算”跳转到订单结算页。  
- **订单**：显示该用户已支付订单列表（按时间或状态排序）。每条订单展示摘要信息（如商品名、主要规格、下单时间、总价等）。点击某条订单进入订单详情页，顶部显示订单状态（制作中、已完成、已取消），详情页列出订单号、付款时间、用餐方式（自提或外送）及具体商品清单等信息。页面底部有“再来一单”按钮，点击后将该订单商品加入购物流程（通常可跳回对应商品或清空当前购物车并添加订单中的商品）。  
- **我的**：显示用户头像、昵称、手机号等信息，下面列出“地址管理”等功能入口，以及退出登录按钮。点击“地址管理”进入地址页，可新增收货地址（姓名、手机号、详细地址）、编辑或删除已保存地址，并将其列为可选项。

页面跳转示意：登录→主界面（tab 切换）→各功能页。商品详情页和订单详情页等为独立路由，可通过编程跳转或路由参数打开。购物车、结算页和地址管理页为功能页面，可通过按钮或链接进入。整套流程用户端通过 UniApp 页面导航、后端数据驱动，各功能间跳转顺畅，最终形成下单支付闭环。  

**商家端**采用后台管理界面布局：首先是管理员登录页，登录成功后进入后台主界面，左侧为侧边导航栏，右侧为内容区。导航栏包含「订单管理」「库存管理」「历史订单」等选项。点击对应菜单会在右侧内容区加载该模块：  
- **订单管理**：展示所有“待处理”订单列表，每条订单显示商品图片、信息、用户、下单时间等；右侧提供“完成”“取消”按钮。点击“完成”后调用后端接口更新订单状态为“已完成”；点击“取消”则更新为“已取消”并触发库存恢复（见下节）。页面可自动刷新展示最新订单。  
- **库存管理**：顶部有搜索框（按商品名），列表显示所有商品条目（名称、分类、价格、库存、上下架状态等）。页面提供“新增商品”按钮，弹出表单窗口填写商品信息（图片、名称、简介、分类、价格、库存、是否上架）；表单可复用于编辑已有商品（点编辑按钮时填充当前数据）。提交表单后调用后台接口创建或更新商品。  
- **历史订单**：类似订单管理列表，但只显示状态为“已完成”或“已取消”的订单，方便查看历史数据。无需额外操作，仅按状态过滤展示。

商家端页面间的主要跳转和交互由 Vue Router 和组件实现：登录→后台首页（侧栏）→各管理模块。页面数据由 Axios 调用后台接口获取并展示，按钮操作触发更新接口。整个后台系统架构简洁，不设计复杂流程，仅实现基本 CRUD 和状态流转功能。

# 数据库表设计建议  
数据库采用 MySQL，共设计 8 张核心业务表：

- 用户表（user）：存储 C 端用户信息，包括用户ID、昵称、手机号、密码（建议哈希存储）、头像URL等。
- 管理员表（admin）：存储商家后台管理员账号，仅用于后台登录。
- 地址表（address）：存储用户收货地址，包括所属用户、收货人姓名、手机号、详细地址等。
- 分类表（category）：存储商品分类，可预置固定分类数据。
- 商品表（product）：存储商品信息，包括分类、名称、简介、图片、价格、库存、是否上架等。
- 购物车表（cart_item）：存储用户当前购物车数据，包括用户ID、商品ID、数量、甜度、温度、杯型、加料等规格信息。
- 订单主表（order_info）：存储订单主体信息，包括订单号、用户、总价、支付时间、用餐方式、地址、订单状态等。
- 订单明细表（order_item）：存储订单中的商品详情，包括商品名称、规格、单价、数量、小计等快照信息。

订单明细保存商品名称、价格、规格等快照数据，即使商品后续修改，也不会影响历史订单显示。

本项目最终数据库结构以 database.sql 文件为唯一标准，不额外设计优惠券、营销、配送等扩展业务表。

# 订单状态与库存规则  
为简化设计，订单状态采用有限的流程：用户提交订单并确认支付后，订单生成时状态为“**制作中（或待制作）**”。管理员处理时可点击“完成”将状态改为“**已完成**”，或“取消”改为“**已取消**”。无须其他中间状态（如“待付款”），支付操作直接创建订单并扣减库存；点击取消即视为订单作废。  
- **库存规则**：用户点击“确认支付”后，后端事务性地创建订单记录并将对应商品的库存数量减去订单数量（若库存不足应校验防止超卖）。如管理员后续取消该订单，应在数据库中将状态改为“已取消”并**恢复**之前扣减的库存数量，使库存保持正确。取消操作需要注意一次性恢复多条订单明细的对应库存。已完成的订单不再变更库存。  
- **自提/外卖逻辑**：订单结算页让用户选择“自提”或“外卖”。若选择自提，只记录用餐方式；若选择外卖，则需要填写地址并在订单记录中关联到用户的某个地址。由于不做实际配送，这里的区别仅影响后端是否要求地址字段。  
- **支付流程**：不接入真实支付，点击“立即支付”后弹出确认框；确认后即视为订单付款成功，执行创建订单及扣库逻辑，取消则返回结算页不执行任何写库操作。无需对接微信支付 SDK，也无后续支付回调流程。  

上述设计确保逻辑清晰：每个订单只有三种状态之一，库存变动仅在创建和取消时处理，避免了复杂的并发补货和分布式事务。符合简化原则，保证初学项目容易实现。

# 复杂度控制与简化建议  
要避免不必要的复杂性，我们将遵循“尽量保持简单”的原则。建议在以下方面简化设计和功能：  
- **技术复杂度**：不使用 Redis、消息队列、微服务等，会用单一数据库和应用程序处理所有逻辑，避免分布式调度和事务一致性问题。前后端通信直接用 REST API，无需引入 Socket、WebSocket、推送 SDK 等。  
- **支付与安全**：不做真正的支付接入，只模拟支付确认；不实现短信验证码、第三方登录等复杂认证方式，采用简单的用户名/密码 + JWT 登录认证即可。
- **购物车管理**：- 购物车采用后端 cart_item 表实现，仅支持基本增删改查，不设计多设备同步、购物车合并、购物车促销等复杂功能。订单创建成功后自动清空对应购物车数据。
- **商品规格处理**：甜度、温度、加料、杯型等选项可设计为固定选项，不用动态扩展字段或表；下单时直接提交选项值字符串。避免设计通用规则引擎或多对多复杂表。  
- **界面功能合并**：比如“立即购买”按钮与购物车结算流程可合并处理：立即购买只是在订单创建接口里以数量 1 直接生成订单，不必再额外设计专门流程；商品编辑与新增可使用同一个表单页面，提交时判断是新增还是更新。  
- **合并模块**：若某些页面或功能明显重复，可合并。例如，订单历史页和订单管理页显示列表的代码可复用；地址新增和编辑使用同一页面模板。  
- **错误和异常**：尽量少抛出过多错误检查，比如库存扣减时只需在下单时校验库存并在同一事务中扣减，不必设计繁琐的超卖补偿策略。系统初期仅面向少量用户练手，可以忽略并发抢购等场景。  
- **前端框架选择**：避免过度依赖第三方库，自主编写简易组件或使用基础 UI 组件库（如 uni-ui、Vant 等）即可，不必自研复杂动画或主题。  

总之，本项目坚持“先将业务跑通，再考虑优化”原则，不急于添加功能。参考 Atlassian “单体式 + 简单交互”的建议，初期聚焦完成核心功能后再逐步改进。

# 开发步骤与优先级建议  
建议按以下顺序逐步开发和调试，以降低难度：  
1. **环境搭建**：安装并创建 Spring Boot 后端骨架项目、Vue 管理后台项目和 UniApp 小程序项目。配置数据库连接（MySQL）和基础依赖，搭建项目目录结构。  
2. **数据库与后端基础**：设计并创建上述数据库表结构。使用 Spring Boot 配置数据源，编写实体类和基本 Repository/DAO。实现通用返回结果封装和异常处理框架。  
3. **用户注册/登录模块**：开发用户注册、登录接口，并在前端完成表单及 API 调用。测试用户验证流程，确保登录后前端能够获得 JWT 身份凭证，并可正常访问需要登录权限的接口。
4. **管理员登录模块**：实现后台管理员账号（可直接在数据库写入），开发登录接口和 Vue 登录页面。登录成功后跳转到后台主框架页面。  
5. **商品与分类基础**：实现获取商品分类和商品列表的 API，前端点餐页显示分类和列表数据。商家后台可先使用模拟数据或简单实现商品查询接口。  
6. **购物车与下单流程**：前端实现商品详情页及购物车交互（选择规格后“加入购物车”刷新浮动条），完成购物车页面或浮层的展示逻辑。开发订单结算页，处理自提/外卖和地址选择，计算总价。后端实现对应接口：创建订单接口应接收用户选项数据，并在数据库生成订单和订单明细记录，扣减库存。  
7. **地址管理**：开发用户地址的增删改查 API 以及前端地址管理页面。保证结算页能调用地址列表接口，让用户选择或添加地址。  
8. **订单展示**：实现用户端订单列表接口 `/api/order/list` 和详情接口 `/api/order/{id}`，前端对应页面展示已下单信息。完善订单状态切换：当管理员更新订单，用户端刷新可见最新状态。  
9. **再来一单功能**：在订单详情页增加“再来一单”按钮。前端点击可将该订单中的商品逐个加入购物车，或直接跳转到相应商品页。后端无需新逻辑，只使用已有的购物车和下单功能。  
10. **商家订单管理**：开发后端订单列表接口（只返回待处理订单）和更新状态接口（完成、取消）。Vue 端调用并显示订单列表，完成/取消按钮触发接口调用。取消时后端需恢复库存并修改订单状态。  
11. **商家库存管理**：开发后端商品增删改查接口。Vue 端实现库存管理页面：显示商品列表，提供搜索，新建和编辑弹窗表单与 API 交互。表单提交触发新增/更新接口，页面刷新列表。  
12. **历史订单页面**：在商家后台增加历史订单视图，复用订单管理逻辑，只显示已完成和已取消的订单（可调用同一个接口并传过滤参数，或前端按状态过滤）。  
13. **测试与完善**：进行端到端测试，模拟用户下单和商家处理流程，检查订单状态、库存变更等是否符合预期。补充异常处理（如库存不足提示）、界面提示（如支付确认框）等细节。优化页面体验，使流程顺畅、错误提示友好。  

每完成一个模块，都可在前后端做简单联调测试，及时修复问题再进行下一个功能。按以上顺序逐步开发，可以保证先打通基础链路，再实现复杂操作。总之，从**用户登录–商品展示–下单支付–订单管理**的流程开始，逐步补充剩余功能，依次攻克，就能构建出满足要求的完整系统。

**参考资料：** 外卖系统设计通常涉及用户、地址、商品、购物车、订单、订单明细等表；为降低复杂度，单体应用结合 REST API 的方式更适合初期项目；同时遵循“最简可行产品”理念，只实现核心功能。以上规划可作为开发的指导蓝本。
